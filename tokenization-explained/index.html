02

How tokens are built: BPE Comment les tokens sont construits : BPE

The algorithm behind GPT-style tokenization is called Byte Pair Encoding (BPE). It starts with individual characters (bytes), then iteratively merges the most frequent adjacent pairs into new tokens — repeating this hundreds of thousands of times on a training corpus.

The result is a vocabulary of ~100,000 tokens for GPT-4. Common English words like the or data become single tokens. Less frequent words like tokenization might be split into token + ization. And a word like the French décomposition might fragment even further.

This is a crucial point: the vocabulary is biased toward English. The training corpus for GPT-4's tokenizer was predominantly English text. French, Arabic, Chinese, and other languages are less efficiently encoded — they need more tokens to express the same information.

L'algorithme derrière la tokenisation GPT s'appelle Byte Pair Encoding (BPE). Il part des caractères individuels (octets), puis fusionne itérativement les paires adjacentes les plus fréquentes en nouveaux tokens — répétant cette opération des centaines de milliers de fois sur un corpus d'entraînement.

Le résultat : un vocabulaire d'environ 100 000 tokens pour GPT-4. Les mots anglais courants comme the ou data deviennent des tokens uniques. Des mots moins fréquents comme tokenisation peuvent être découpés en fragments. Et un mot arabe courant peut se fragmenter encore davantage.

Point crucial : le vocabulaire est biaisé vers l'anglais. Le corpus d'entraînement du tokeniseur GPT-4 était majoritairement en anglais. Le français, l'arabe, le chinois et d'autres langues sont encodés moins efficacement — ils nécessitent plus de tokens pour exprimer la même information.

Numbers & Tokens

The model doesn't see "27" and "83" as numbers — it sees tokens, the same way it sees words. There's no built-in "digit" type. When you ask it to add 27 + 83, it has to learn arithmetic across token boundaries that shift unpredictably depending on context. That's why 3-digit addition works but 8-digit multiplication often fails.

Le modèle ne voit pas "27" et "83" comme des nombres — il voit des tokens, exactement comme il voit des mots. Il n'y a pas de type "chiffre" intégré. Quand on lui demande d'additionner 27 + 83, il doit apprendre l'arithmétique à travers des frontières de tokens qui changent imprévisiblement selon le contexte. C'est pourquoi l'addition à 3 chiffres fonctionne mais la multiplication à 8 chiffres échoue souvent.

04

The language tax La taxe linguistique

LLM APIs charge per token. If your language is poorly represented in the tokenizer's training data, the same sentence costs you more — and the model understands it less efficiently. This is a real, measurable inequality.

07

What this actually changes Ce que ça change concrètement

Tokenization is the most underexplained part of working with LLMs. Here's what understanding it actually unlocks:

Context windows are measured in tokens, not words. GPT-4's 128k context isn't 128,000 words — it's 128,000 token slots. A French document uses those slots ~20% faster than an equivalent English one.

Prompt injection often exploits token boundaries. Attackers use unusual tokenization patterns (like Unicode lookalikes, unusual whitespace, or token splits) to bypass content filters that work at the word level.

Code is surprisingly cheap. Python and JavaScript have lots of common, repetitive sequences that become single tokens. A function definition costs far fewer tokens than you might expect — which is why coding assistants are so economically viable.

Repeating a word isn't the same as saying it once. If you write "very very very" in a prompt, the model doesn't parse emphasis the way humans do — it processes three sequential tokens, each shifting the probability distribution slightly. Knowing this helps you communicate intent more clearly.

La tokenisation est la partie la moins expliquée du travail avec les LLMs. Voici ce que sa compréhension débloque réellement :

Les fenêtres contextuelles se mesurent en tokens, pas en mots. Le contexte de 128k de GPT-4 ne représente pas 128 000 mots — c'est 128 000 emplacements de tokens. Un document français consomme ces emplacements environ 20% plus vite qu'un équivalent anglais.

L'injection de prompts exploite souvent les frontières de tokens. Les attaquants utilisent des patterns de tokenisation inhabituels (caractères Unicode similaires, espaces inhabituels, découpes de tokens) pour contourner les filtres de contenu qui opèrent au niveau des mots.

Le code est étonnamment peu coûteux. Python et JavaScript ont beaucoup de séquences courantes et répétitives qui deviennent des tokens uniques. Une définition de fonction coûte bien moins de tokens qu'on ne le pense — c'est pourquoi les assistants de code sont économiquement viables.

Répéter un mot n'est pas la même chose que le dire une fois. Si vous écrivez "très très très" dans un prompt, le modèle ne perçoit pas l'emphase comme un humain — il traite trois tokens séquentiels, chacun déplaçant légèrement la distribution de probabilité.

Bottom line:En résumé : You don't need to count tokens manually in your prompts. But knowing that tokens are the unit of computation — not words, not characters, not sentences — changes how you think about what the model is actually doing.Vous n'avez pas besoin de compter les tokens manuellement dans vos prompts. Mais savoir que le token est l'unité de calcul — pas les mots, pas les caractères, pas les phrases — change la façon dont vous comprenez ce que fait réellement le modèle.
