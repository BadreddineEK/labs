01

The trap that fools everyone Le piège qui trompe tout le monde

You train a model on historical sales data. You evaluate it. RMSE looks great — better than any baseline. You deploy. It falls apart in production.

This is not bad luck. It's a structural problem: your test set saw the future during training. The model didn't learn patterns — it memorized outcomes it was never supposed to see.

In standard machine learning, shuffling data before splitting is fine. In time series, it's catastrophic. Time has a direction. Training on data that includes information from after your forecast horizon is the definition of leakage.

Vous entraînez un modèle sur des données de ventes historiques. Vous l'évaluez. Le RMSE est excellent — meilleur que n'importe quelle baseline. Vous déployez. Il s'effondre en production.

Ce n'est pas de la malchance. C'est un problème structurel : votre ensemble de test a vu le futur pendant l'entraînement. Le modèle n'a pas appris des patterns — il a mémorisé des résultats qu'il n'était pas censé voir.

En machine learning classique, mélanger les données avant de les diviser est correct. En séries temporelles, c'est catastrophique. Le temps a une direction. S'entraîner sur des données qui incluent des informations postérieures à votre horizon de prévision, c'est la définition du leakage.

The rule:La règle : In a time series, point T in your test set must never influence any computation that happens before T in your training pipeline — not scaling, not feature engineering, not imputation. Never.Dans une série temporelle, le point T de votre test ne doit jamais influencer aucun calcul qui se produit avant T dans votre pipeline d'entraînement — pas le scaling, pas le feature engineering, pas l'imputation. Jamais.

The split date is just the most obvious form. In practice, leakage sneaks in at every step of the pipeline. Here are the three forms that kill forecasting models in production.

La date de coupure est juste la forme la plus évidente. En pratique, le leakage s'infiltre à chaque étape du pipeline. Voici les trois formes qui détruisent les modèles de forecasting en production.

The subtlest form is feature leakage. Imagine you build a feature like "rolling 7-day average of sales". If you compute this on the entire dataset before splitting, the rolling average for day 100 includes data from days 101–107. The model trains on a feature that summarizes the future.

La forme la plus subtile est le feature leakage. Imaginez que vous construisiez une feature comme "moyenne mobile sur 7 jours des ventes". Si vous la calculez sur l'ensemble du dataset avant de le diviser, la moyenne mobile du jour 100 inclut des données des jours 101 à 107. Le modèle s'entraîne sur une feature qui résume le futur.

04

The fix: walk-forward validation La solution : la validation walk-forward

Standard k-fold cross-validation is broken for time series. It shuffles folds, so training folds include future data relative to validation folds. The correct approach is walk-forward validation (also called time-series split): you always train on the past, validate on the immediate future, and slide forward.

La validation croisée k-fold standard est cassée pour les séries temporelles. Elle mélange les folds, donc les folds d'entraînement contiennent des données futures par rapport aux folds de validation. L'approche correcte est la validation walk-forward (aussi appelée time-series split) : on s'entraîne toujours sur le passé, on valide sur le futur immédiat, et on avance.

Chaque fold vous donne une estimation réaliste de la performance du modèle sur des données futures non vues. Vous obtenez plusieurs valeurs de RMSE au lieu d'une seule — et leur variance dit aussi quelque chose : un modèle avec des scores de fold très incohérents est fragile, indépendamment de la moyenne.

05

The sneaky one: scaling leakage Le plus sournois : le leakage de normalisation

Here's one that even experienced practitioners miss. You use StandardScaler or MinMaxScaler before splitting. You fit the scaler on the full dataset — which includes test data. The mean and standard deviation used for normalization now contain information about the future. The leak is subtle, but the test set is compromised.

En voici une que même les praticiens expérimentés ratent. Vous utilisez StandardScaler ou MinMaxScaler avant de diviser les données. Vous fittez le scaler sur le dataset complet — qui inclut les données de test. La moyenne et l'écart-type utilisés pour la normalisation contiennent maintenant des informations sur le futur. La fuite est subtile, mais l'ensemble de test est compromis.

✗ Wrong:
scaler.fit(X_all) → split → transform(X_train), transform(X_test)

✓ Correct:
split first → scaler.fit(X_train) → transform(X_train), transform(X_test)

The same rule applies to every preprocessing step: imputation, encoding, outlier removal, PCA. Fit on train only. Apply to test. The test set must remain a genuine stranger to all fitting operations.

La même règle s'applique à chaque étape de prétraitement : imputation, encodage, suppression des outliers, PCA. Fittez uniquement sur le train. Appliquez au test. L'ensemble de test doit rester un véritable inconnu pour toutes les opérations de fitting.

06

How to diagnose leakage in your own model Comment diagnostiquer le leakage dans votre propre modèle

If you suspect leakage, here's a practical diagnostic. None of these alone is conclusive, but together they paint a clear picture.

Si vous suspectez un leakage, voici un diagnostic pratique. Aucun de ces éléments seul n'est concluant, mais ensemble ils donnent une image claire.

The most reliable test: deliberately introduce a random target value into your test set, then retrain. If your "test RMSE" stays suspiciously good, the training data has seen the test targets. The model is interpolating, not forecasting.

Le test le plus fiable : introduire délibérément une valeur cible aléatoire dans votre test, puis ré-entraîner. Si votre "RMSE de test" reste suspicieusement bon, les données d'entraînement ont vu les cibles de test. Le modèle interpole, il ne prévoit pas.

07

What this actually changes in practice Ce que ça change concrètement en pratique

Leakage inflates metrics, not understanding. A model with leakage has learned the shape of the test data — not the underlying process. Deploy it 6 months later with a different trend regime and it will be confidently wrong.

Business decisions built on leaked models are dangerous. Inventory forecasts, staffing plans, drug trial outcome predictions — all downstream of a metric that was fabricated. The gap between reported RMSE and production RMSE is a hidden liability.

A higher RMSE from a clean model beats a low RMSE from a leaked one, every time. When you fix leakage, your metric will get worse on paper. That's not a regression — it's honesty. The number finally means something.

Le leakage gonfle les métriques, pas la compréhension. Un modèle avec leakage a appris la forme des données de test — pas le processus sous-jacent. Déployez-le 6 mois plus tard avec un régime de tendance différent et il sera confiant dans l'erreur.

Les décisions métier basées sur des modèles contaminés sont dangereuses. Prévisions de stocks, plans de staffing, prédictions de résultats d'essais cliniques — tous en aval d'une métrique fabriquée. L'écart entre le RMSE rapporté et le RMSE en production est un passif caché.

Un RMSE plus élevé issu d'un modèle propre bat chaque fois un RMSE bas d'un modèle contaminé. Quand vous corrigez le leakage, votre métrique se dégrade sur le papier. Ce n'est pas une régression — c'est de l'honnêteté. Le chiffre veut enfin dire quelque chose.

One rule to remember:Une règle à retenir : If building your features requires knowing any value from t+1 or later, you have leakage. No exceptions. No "it's just a small effect". Time only flows forward.Si construire vos features nécessite de connaître une valeur à partir de t+1 ou plus tard, vous avez un leakage. Pas d'exceptions. Pas de "c'est juste un petit effet". Le temps ne s'écoule que vers l'avant.
