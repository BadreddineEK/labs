Streamlit Under the Hood

You write a Python script. A user clicks a button. The entire script re-runs. That's Streamlit. But what exactly happens — line by line, state by state — is what most tutorials never explain.

5Concepts
~7 minRead
Apr 2026Published
01

The execution model

Most web frameworks are event-driven: a button click fires a specific callback, and only that callback runs. Streamlit works differently. Every user interaction — a slider move, a button click, a text input — triggers a full re-execution of your entire Python script, from top to bottom.

This is the single most important thing to understand about Streamlit. It feels counterintuitive at first, but it's what makes Streamlit so simple to write: there's no event system, no callbacks to wire up, no component state to synchronize. The script is the UI.

🔁
Full re-run on interaction
Every widget interaction reruns the script. Streamlit diffs the output and updates only what changed in the browser.
📜
Script = UI declaration
Your script is a declarative description of the UI state at the moment of execution. No templates, no JSX, just Python top-to-bottom.
⚡
Smart diffing
Streamlit's frontend React app receives a delta of changes from the server — only the widgets that changed are updated in the DOM.
02

Watch a re-run happen

Here's a minimal Streamlit app. Click the button or move the slider to trigger a re-run and watch which lines execute, which are cached, and what the output looks like after each run.

app.py 
import streamlit as st

@st.cache_data
def load_data():
    # expensive — only runs once
    return [1, 2, 3, 4, 5]

data = load_data()          # cached after first run
n = st.slider("Multiplier", 1, 10, 3)

if st.button("Compute"):
    result = [x * n for x in data]
    st.write(f"Result: {result}")

03

Session state

Since the script re-runs completely, normal Python variables are reset on every interaction. st.session_state is the escape hatch — a dictionary that persists across re-runs within a user session. It's the only memory a Streamlit app has between executions.

What it is
How to use it
Patterns
type
Dict-like object that persists across re-runs. Lives in server memory, scoped to one browser session.
lifetime
Created when the user first connects. Destroyed when the tab closes or the session times out (default: a few minutes of inactivity).
access
st.session_state["key"] or st.session_state.key — both work. Initialize with .get() or check with "key" in st.session_state.
widgets
Widgets with a key= argument automatically sync their value into st.session_state — you can read and write them.
initialize
if "count" not in st.session_state: st.session_state.count = 0 — always guard initialization to avoid overwriting on re-run.
increment
st.session_state.count += 1 — modify inside a callback or directly in the script body after checking the trigger widget.
callbacks
Use on_change= / on_click= to run a function before the re-run. State updates inside callbacks are applied first, then the script reruns.
reset
del st.session_state["key"] or st.session_state.clear() — use sparingly; prefer explicit resets over full clears.
counter
Classic: st.session_state.count += 1 on button click — persists the count across re-runs without a database.
multi-step form
Store st.session_state.step (0, 1, 2…) and conditionally render sections. Navigate forward/back by updating step.
chat history
st.session_state.messages = [] then append on each exchange. Render with st.chat_message() in a loop.
toggled panels
Store st.session_state.show_advanced = False; flip on button click. Conditionally render the panel.
04

Caching — the performance lever

Since the entire script re-runs on every interaction, expensive operations — loading a large CSV, calling an API, training a model — would run on every single re-run. Caching is Streamlit's solution: decorators that memoize function output so it's only recomputed when the inputs change.

@st.cache_data
For data objects (DataFrames, lists, dicts)
Serializes and caches the return value. Each session gets its own copy. Safe for mutable objects.
@st.cache_resource
For global resources (DB connections, models)
Caches the object itself — shared across all sessions. Use for heavy singletons like ML models or DB connectors.
Click a button to simulate function calls…

Cache invalidation happens automatically when the function's arguments change, when the source code changes, or when you explicitly call st.cache_data.clear(). You can also set a ttl= (time-to-live) to expire cache entries after a set duration.

05

Streamlit vs React

Streamlit's architecture looks nothing like a standard React app — yet internally, its frontend is built in React. Understanding the difference clarifies why Streamlit is fast to write but has real architectural tradeoffs at scale.

St
Streamlit
Rendering model
Full script re-run on every interaction. Server-side Python generates a delta, pushed to frontend via WebSocket.
State
st.session_state — server-side, per-session dict. No client-side state management needed.
Interactivity
Widgets declaratively defined in Python. All logic lives in the script.
Best for
Internal data tools, ML demos, rapid prototyping, dashboards with heavy Python backends.
Re
React
Rendering model
Component-level re-renders triggered by state/prop changes. Virtual DOM diffing. No server round-trip for UI updates.
State
useState, useReducer, Context, Redux — client-side, granular, explicit. Full control, more boilerplate.
Interactivity
Event handlers on individual components. Highly granular control over what updates and when.
Best for
Production web apps, complex UIs, high-traffic public products, real-time experiences.

What to remember

Streamlit in five sentences.

  1. Every interaction triggers a full script re-run — this is Streamlit's core contract, not a bug.
  2. Widgets are declarative: their position in the script determines their order in the UI, their return value is their current state.
  3. st.session_state is the only memory that survives a re-run — use it for counters, form steps, chat history, and toggle states.
  4. @st.cache_data and @st.cache_resource prevent expensive functions from re-running on every interaction.
  5. Streamlit trades control for speed: it's the fastest path from a Python idea to a shareable data app, at the cost of full frontend flexibility.
